12/6/18

Big news: the battery management system is most of the way done. I have the prototype built and working mostly. Its an absolute mess and im not using the final components so in the end it will be using a relay board so I can control 8 different battery cells and it will be nice and clean with an arduino nano so that everything is nice and compact. I'll be uploading schematics later but its a way of dynamically changing battery configuration so that when charging the batteries are all in a parallel configuration and thus self balance ensuring that the batteries can't drift too far some one another in terms of voltage. Then when the system needs to run the configuration changes so that they cells are in parallel and produce a higher voltage needed for running all of the other circuitry. the other cool part is that because the batteries fail to parallel, if the system loses power for any reason the batteries drop back to the parallel configuration and the likelihood of something bad happening drops dramatically.

12/28/18

So, on my instagram I have a very candid video series documenting the work I've been doing over the past few days. In a written and more professional form I will do some more explaining. I am using four HC-sr04 ultrasonic sensors to figure out where walls are. They emmit an ultrasonic pulse at 40k Hz, and listen and we measue how long it takes for it to hear that echo, from that we can calculate the time of flight and we know the distance that it reports through some simple algebra. There is a pre-made library called GPIO zero which I think I've mentioned before, in a short form: it makes interfacing with sensors and stuff really easy. Easy until theres bad data, and lots of it. With the pre-made gipo zero system I am getting loads of garbage data. I have been running tests using different resistors and some algebra to average the data. I have come to the conclusion that there is some black magic at work... or the gpio zero class for the sensors doesn't work as it really should. Running a range of tests between 10,000 and 100,000 measurements with removing bad data and averaging and everything I could think of to remove bad data and collect better data taught me a lot. A lot of the time was taken up with trying to average data, going from 100,000 measurements to 10,000 measurements cleaned up data and the time it took dropped 66% but that includes removing math too. So the data doesn't benefit from averaging I was seeing larger deltas in the data. One good measurement is a lot better than 10 ok/bad measurements. Now the trick is to get the sensors to play well when they are setup in an array. I guess I could stagger the measurements so that the pulses dont overlap and only measure stuff up to a certian distance and anything outside of that is just neglected. Maybe have an algorithm that will identify bad data and get new data.

SQL auto deploy:

During the summer of 2018 we built a tool for automated sql server deployment. The process for using this utility is, push all of the sql files to the VSTS folder and start the utility. The utility will figure out what files need to be run. It will then run and log every file that needs to be run while cathcing and alerting octopus if there was an error so a roll back can be performed. Then all of that log information is put on a website so that any developer in the company can view the logs and fix issues without needing to be told by a superior what went wrong. This tool was also tested to run 200 sql comand files and log everything in under a minute.
